# Руководство по защите страниц

## Обзор

Реализована система защиты страниц через `proxy.ts`, которая контролирует доступ к различным разделам сайта в зависимости от статуса авторизации и роли пользователя.

## Логика защиты

### 1. **Landing Page (`/`)**

**Неавторизованные пользователи:**
- ✅ Видят landing page

**Авторизованные обычные пользователи:**
- 🔄 Автоматически перенаправляются в `/profile`

**Авторизованные админы:**
- 🔄 Автоматически перенаправляются в `/admin/dashboard`

### 2. **Страницы авторизации**

Страницы:
- `/login` - вход
- `/register` - регистрация
- `/verify-email` - верификация email
- `/forgot-password` - забыли пароль
- `/reset-password/verify` - ввод кода сброса
- `/reset-password/new` - новый пароль

**Неавторизованные пользователи:**
- ✅ Имеют доступ ко всем страницам

**Авторизованные обычные пользователи:**
- 🔄 Автоматически перенаправляются в `/profile`

**Авторизованные админы:**
- 🔄 Автоматически перенаправляются в `/admin/dashboard`

### 3. **Страница профиля (`/profile`)**

**Неавторизованные пользователи:**
- 🔄 Автоматически перенаправляются на `/login`

**Авторизованные обычные пользователи:**
- ✅ Имеют доступ

**Авторизованные админы:**
- 🔄 Автоматически перенаправляются в `/admin/dashboard`

### 4. **Админские страницы (`/admin/*`)**

#### Страницы входа (`/admin/login`, `/admin/verify`)

**Неавторизованные пользователи:**
- ✅ Имеют доступ

**Авторизованные админы:**
- 🔄 Автоматически перенаправляются в `/admin/dashboard`

**Авторизованные обычные пользователи:**
- ✅ Имеют доступ (для входа как админ)

#### Защищенные админские страницы (`/admin/dashboard` и др.)

**Неавторизованные пользователи:**
- 🔄 Автоматически перенаправляются на `/admin/login`

**Авторизованные обычные пользователи:**
- 🔄 Автоматически перенаправляются на `/` (нет прав)

**Авторизованные админы:**
- ✅ Имеют доступ

## Технические детали

### Файл: `proxy.ts`

```typescript
export async function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const token = request.cookies.get('auth_token')?.value;
  
  let user = null;
  if (token) {
    user = await verifyToken(token);
  }
  
  // Логика защиты страниц...
}
```

### Проверка авторизации

Используется JWT токен из cookie `auth_token`:
- Токен проверяется функцией `verifyToken()` из `lib/auth.ts`
- Возвращает объект с `userId`, `email`, `role`
- Если токен невалиден, возвращает `null`

### Роли пользователей

- `user` - обычный пользователь
- `admin` - администратор
- `manager` - менеджер (пока не используется)

## Примеры сценариев

### Сценарий 1: Обычный пользователь

1. **Заходит на `/`** → Видит landing page
2. **Регистрируется** → `/register` → `/verify-email`
3. **Подтверждает email** → Перенаправление в `/profile`
4. **Пытается зайти на `/`** → Автоматически перенаправляется в `/profile`
5. **Пытается зайти на `/login`** → Автоматически перенаправляется в `/profile`

### Сценарий 2: Админ

1. **Заходит на `/admin/login`** → Вводит credentials
2. **Подтверждает через Telegram** → `/admin/verify`
3. **Успешная авторизация** → Перенаправление в `/admin/dashboard`
4. **Пытается зайти на `/`** → Автоматически перенаправляется в `/admin/dashboard`
5. **Пытается зайти на `/profile`** → Автоматически перенаправляется в `/admin/dashboard`

### Сценарий 3: Неавторизованный пользователь

1. **Заходит на `/`** → Видит landing page
2. **Пытается зайти на `/profile`** → Автоматически перенаправляется на `/login`
3. **Пытается зайти на `/admin/dashboard`** → Автоматически перенаправляется на `/admin/login`

## Исключения из защиты

Следующие пути **не проверяются** middleware:
- `/_next/*` - статические файлы Next.js
- `/api/*` - API endpoints (имеют свою защиту)
- `/public/*` - публичные файлы
- Файлы с расширениями (`.svg`, `.png`, `.jpg`, и т.д.)

## Настройка matcher

```typescript
export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico).*)'],
};
```

Это гарантирует, что middleware применяется только к нужным маршрутам.

## Безопасность

### JWT токен
- ✅ Хранится в httpOnly cookie
- ✅ Защищен от XSS атак
- ✅ Срок действия: 7 дней
- ✅ Проверяется на каждом запросе

### Проверка роли
- ✅ Роль проверяется на сервере
- ✅ Невозможно подделать роль на клиенте
- ✅ Админы не могут заходить на пользовательские страницы
- ✅ Пользователи не могут заходить на админские страницы

## Добавление новых защищенных страниц

### Для страниц авторизации:
```typescript
const authPages = [
  '/login',
  '/register',
  '/new-auth-page', // Добавить сюда
];
```

### Для защищенных пользовательских страниц:
```typescript
if (pathname.startsWith('/new-protected-page')) {
  if (!user) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
  return NextResponse.next();
}
```

### Для админских страниц:
```typescript
if (pathname.startsWith('/admin/new-page')) {
  if (!user) {
    return NextResponse.redirect(new URL('/admin/login', request.url));
  }
  if (user.role !== 'admin') {
    return NextResponse.redirect(new URL('/', request.url));
  }
  return NextResponse.next();
}
```
